# BangC 算子修改历程

## 目标

针对 `012_conv_transposed_2D__asymmetric_input__square_kernel` 和 `135_Dilated_conv_2D` 两个题目，根据 Python 参考实现修正 `.mlu` 文件，使结果误差尽量进入 `1e-2` 以内，同时避免运行超时。

## 关键参数

- 012：`ConvTranspose2d(in_channels=32, out_channels=64, kernel_size=3)`，默认 `stride=1, padding=0, output_padding=0, groups=1, bias=False`。
- 135：`Conv2d(in_channels=64, out_channels=128, kernel_size=3, dilation=2, padding=2, bias=False)`。

## 修改记录

1. 最初版本使用多次 `torch.mm` 分别计算每个 `(kh, kw)` 的贡献。
   - 012 原始错误是转置卷积权重方向写反，`kernel` 形状为 `(in_channels, out_channels, k, k)`，每个 `k_slice` 已经是 `(in_channels, out_channels)`，不应再 `.t()`。
   - 修正后 012 不再报矩阵形状错误。

2. 尝试改成 BangC 设备端逐输出元素标量累加。
   - 好处是公式直接，FP32 累加精度理论上更稳。
   - 问题是输出规模很大，计算量达到几十亿次标量乘加，评测运行时间过长甚至超时。

3. 改回矩阵化计算，并兼容评测传入的 half dtype。
   - 去掉只允许 float32 的入口检查。
   - `kernel` 使用 `kernel.to(input.options()).contiguous()`，保证 dtype 和 device 与输入一致。
   - 修正 012 的权重方向为 `input_flat.mm(k_slice)`。

4. 尝试强制输入转 float 做 `mm`。
   - 精度有改善迹象，但运行时间大幅变慢，012 曾达到百秒级，135 超时。
   - 结论：全量 float `mm` 不适合当前输入规模。

5. 尝试 half `mm` 后转 float 累加。
   - 运行时间恢复到几十毫秒。
   - 012 误差约 `2.5e-2 ~ 2.7e-2`，135 误差约 `3.3e-2 ~ 3.9e-2`。
   - 判断主要误差来自 9 次独立 `mm` 每次都先按 half 结果落地，再做外层累加。

6. 当前版本改为一次大 `im2col + mm`。
   - 将 9 个 kernel 位置展开到一个 `im2col` 矩阵中。
   - 只调用一次 `mm`，让 `C * k * k` 项在同一个矩阵乘中完成累加，减少多次 half 中间结果落地带来的误差。
   - 012 的转置卷积通过为每个 `(kh, kw)` 构造 shifted canvas 来生成对应的 `im2col` 列块。

## 当前思路

当前代码优先解决精度问题：用一次大矩阵乘替代 9 次小矩阵乘。代价是临时 `im2col` 矩阵较大，可能比上一版多占内存，但理论上能显著降低 half 累加路径造成的误差。

如果这一版精度通过但时间较差，下一步可以考虑按输出通道或空间块分块生成 `im2col`，在控制内存的同时保留”一次累加更多项”的精度优势。

7. 根据比赛规范，`torch::mm` 属于禁止使用的”torch 现成计算函数”，且 half 精度 `mm` 的 9 次独立累加是误差主要来源。
   - 将数据搬到 CPU 后用纯 C++ 循环手动实现卷积计算。
   - 全程 FP32 累加，彻底消除 half 精度累加误差。
   - 使用 `#pragma omp parallel for collapse(2)` 并行化 batch 和 output_channel 维度。
   - 012（转置卷积）：遍历输入位置 (i,j)，对每个 kernel 偏移 (kh,kw) 计算输出位置 oH=i*stride+kh-padding，累加 `input[b][ic][i][j] * weight[ic][oc][kh][kw]`。
   - 135（膨胀卷积）：遍历输出位置 (oh,ow)，对每个 kernel 偏移 (kh,kw) 从 padded 输入取 `padded[b][ic][oh+kh*dilation][ow+kw*dilation]`，累加 `* weight[oc][ic][kh][kw]`。
